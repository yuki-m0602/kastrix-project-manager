## 確認ルール

### 確認なしでOKな操作
- ファイルの読み取り・検索
- 既存テストの実行
- 仕様書・設計書の読み込み

### 必ず事前確認が必要な操作
- 作業依頼を受けた場合 → どう理解したかを説明してから着手する
- 問題点・改善点に気づいた場合 → 提案を示して承認を得てから実装する
- 新しい外部ライブラリの追加 → 理由と代替案を提示してから追加する
- 既存ファイルの削除・リネーム → 内容を説明してから実施する
- スコープ外の機能実装 → 提案して承認を得てから実装する
- 大規模なリファクタリング → `agent_docs/refactoring.md` に従う
- git push / デプロイ操作 → 確認を取ってから実行する

---

## コーディングルール

- 既存コードのスタイル・命名規則に従う
- フォーマッタ設定があればそれに従う（自己判断しない）
- 型定義がある言語では型を省略しない
- DRY原則を徹底する。同じロジック・処理は必ず共通化する
- 1ファイルが肥大化する場合はファイルを分割する（ファイル数増加より肥大化を避ける）
