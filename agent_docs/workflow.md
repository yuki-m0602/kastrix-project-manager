# 作業フロー

## 進め方の原則

設計確認 → プロトタイプ → 機能単位で肉付け → レビュー の順で進める。

---

## Phase 1: 設計確認

1. `specs/` `plans/` の関連ドキュメントを読む
2. `mockup/` にUIモックアップがあれば確認する
3. スコープ（何を作るか・何を作らないか）と完了定義をユーザーと合意する
4. 矛盾・不明点があれば実装前にユーザーに質問する

仕様書が存在しない場合は、ユーザーの指示を仕様として扱う。

---

## Phase 2: 実装

プロトタイプ優先。「まず動く状態」を目指し、細部の作り込みは後回し。

```
[ ] コアロジック（最小限で動く状態）
[ ] バリデーション
[ ] エラーハンドリング
[ ] API / データ接続
[ ] UI調整
[ ] パフォーマンス改善
```

- 1機能ずつ実装・動作確認してから次へ進む
- 「触れる・見れる」状態になったらユーザーに確認を取る
- 仕様変更が発生したら `specs/` も合わせて更新する

---

## Phase 3: レビュー

実装完了後、以下を確認してユーザーに報告する：

- セキュリティ上の懸念点
- パフォーマンス上の懸念点
- 仕様書との整合性
- テスト結果

仕様とのズレがあれば指摘し、仕様書の更新も合わせて提案する。
