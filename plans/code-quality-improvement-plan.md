# Kastrix コード品質改善方針

> 作成日: 2026-03-16  
> 参照: `agent_docs/refactoring.md`（実施時は必ず確認）

---

## 1. 現状サマリ

| カテゴリ | 主な課題 |
|----------|----------|
| **DRY** | tasks.rs の同期ブロック重複、team.rs のブロードキャスト・招待・権限チェックの重複 |
| **SRP** | settings.js が複数責務を抱えている |
| **セキュリティ** | tasks.js / projects.js でユーザー入力を innerHTML にそのまま挿入（XSS リスク） |
| **エラー処理** | db.rs の expect、フロントの alert 多用 |
| **設計** | グローバル変数、マジック値、言語混在 |
| **テスト** | ユニットテストがほぼない |
| **Lint** | ESLint・rustfmt の設定がない |
| **DB** | ALTER TABLE が db.rs に直書き、マイグレーション管理が未整備 |

---

## 2. 改善方針（4フェーズ）

### Phase 0: 土台（事前準備）

| # | 項目 | 対象 | 内容 |
|---|------|------|------|
| 0-1 | テスト基盤 | src-tauri | 重要コマンド（get_projects, get_tasks 等）からテストを追加 |
| 0-2 | Lint 導入 | 全体 | ESLint（JS）、rustfmt / clippy（Rust）の設定 |
| 0-3 | DB マイグレーション | db.rs | ALTER TABLE をマイグレーションスクリプトに切り出し、バージョン管理 |

**目安**: 1〜2日

---

### Phase 1: セキュリティ・安定性（最優先）

| # | 項目 | 対象 | 内容 |
|---|------|------|------|
| 1 | XSS 対策 | tasks.js, projects.js | title, name, assignee, description, path 等を escapeHtml でエスケープ |
| 2 | パニック防止 | db.rs | expect を `?` や map_err に置き換え、呼び出し元でエラーを返す |

**目安**: 1〜2日

---

### Phase 2: DRY・保守性（中期的）

| # | 項目 | 対象 | 内容 |
|---|------|------|------|
| 3 | 同期ブロック共通化 | tasks.rs | `maybe_broadcast_task_update()` を導入し、4コマンドから呼び出す |
| 4 | ブロードキャスト共通化 | team.rs | `broadcast_json_payload()` を導入し、各ブロードキャスト関数を共通化 |
| 5 | 招待・権限ロジック共通化 | team.rs | `insert_invite_code`, `can_approve_or_reject`, `get_my_endpoint_id`, `is_host` をヘルパー化 |
| 6 | API 呼び出しラッパー | api.js | `invokeWithDefault(cmd, args, default)` を導入し、try-catch の重複を削減 |

**目安**: 3〜5日

---

### Phase 3: 構造・一貫性（長期的）

| # | 項目 | 対象 | 内容 |
|---|------|------|------|
| 7 | 責務の分離 | settings.js | 監視ディレクトリ / AI / チーム / IDE を別モジュールに分割 |
| 8 | エラー表示の統一 | フロント全体 | alert を減らし、トーストやインライン表示を共通化 |
| 9 | 定数・マジック値の整理 | 全体 | タイムアウト、色コード、sync_mode 等を定数ファイルに集約 |
| 10 | 言語の統一 | メッセージ | 日本語 / 英語のどちらかに統一 |
| 11 | mockup 整理 | mockup/ | プロトタイプ用途に限定するか、ui に統合するか方針を決める |

**目安**: 1〜2週間

---

## 3. 実施ルール

1. **事前**: 各変更前に `agent_docs/refactoring.md` に従い、対象・理由・影響範囲を整理してから着手する
2. **段階**: Phase 0 → 1 → 2 → 3 の順で進める
3. **テスト**: 各変更後に `cargo test` と手動確認を実施する
4. **粒度**: 1項目ずつ実施し、まとめて変更しない

## 5. 推奨着手順

1. Phase 0-1: テスト基盤
2. Phase 0-2: Lint 導入
3. Phase 1-1: XSS 対策（tasks.js / projects.js）
4. Phase 1-2: db.rs の expect 削除
5. Phase 2-3: tasks.rs の同期ブロック共通化
6. 以降は Phase 2 → Phase 3 の順で進める

- **本格的な i18n**: 現状は言語の統一で十分
