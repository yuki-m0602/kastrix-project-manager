 Kastrix — dev 版と exe 版の起動経路（このリポジトリ前提の説明）
========================================================================

■ 経路の対比（起動してから UI / Rust / データまで）

  [A] DEV（例: dev.bat）
      1) 別ウィンドウ: npm run css:watch
         - ui\src\styles.css を監視し ui\vendor\tailwind.css を都度書き換え
      2) このプロセス: npx tauri dev
         - Cargo workspace ルートで target\debug\kastrix.exe をビルドして実行
         - Rust は debug プロファイル（最適化弱め、assert 等の挙動が release と違うことがある）
         - tauri.conf の beforeDevCommand は空、devUrl も無いため
           WebView は HTTP ではなく frontendDist（相対 ../ui）の静的ファイルを読む
         - 実行中も ui フォルダの中身（特に tailwind.css）が変わりうる
      3) アプリデータ・DB・iroh 鍵などは Tauri の app_data_dir 配下
         （identifier は com.kastrix.desktop — バッチ実行時に推定パスを表示）
      4) 標準出力と eprintln は、この起動したコンソールに出る

  [B] RELEASE exe（例: build.bat のあと target\release\kastrix.exe を直接起動）
      1) ユーザーが実行するのは release ビルド済みの kastrix.exe
      2) Rust は release プロファイル（Cargo.toml の strip, LTO, panic=abort 等）
      3) UI はビルド時点の ../ui がバンドルに取り込まれたもの（実行中にリポジトリの ui を
         書き換えても exe 内の資産は自動では変わらない）
      4) コンソール無しウィンドウの場合、ログはファイルやデバッガでしか見えない
      5) カレントディレクトリは起動方法で変わる（エクスプローラーからだと exe の場所や
         System32 などになり、相対パス前提のコードがあると dev と結果が変わることがある）

  [C] インストーラ版（NSIS / MSI で配布したもの）
      - exe の実体パスが target\release\ とは別（Program Files 等）
      - 上記 B に加え、インストール場所・権限・更新タイミングの差が乗る

■ package.json の npm run dev について
  live-server で ui をポート 1420 で配信するスクリプトだが、dev.bat はこれを使っていない。
  現在の dev 経路は「ファイル直読み」と「tailwind watch」。

■ 同じ version でも挙動が分かれやすいポイント
  - debug と release のコード生成・最適化・panic 方針
  - UI が「開発時は常に最新の ui フォルダ」か「ビルド時スナップショット」か
  - カレントディレクトリと環境変数（PATH、プロキシ、TAURI 系など）
  - インストール版とローカル exe では実行パス・権限が異なる
