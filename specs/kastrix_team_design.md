# Kastrix — チーム機能 設計ドキュメント

2026年3月 — v1.0

---

## 1. 概要

現状のKastrixはローカル・個人向けのデスクトップアプリである。本ドキュメントでは、チームメンバー間でプロジェクト・タスク情報をリアルタイムに同期するチーム機能の設計をまとめる。

**設計の基本方針：**

- **サーバーレス** — 専用サーバーを持たずP2Pで同期する
- **ローカルファースト** — オフラインでも全機能が使える
- **開発者向け** — セキュリティと透明性を重視する

---

## 2. チーム機能の方向性

| カテゴリ | 機能例 | 優先度 |
|----------|--------|--------|
| リアルタイム共有・同期 | タスク変更の即時同期、複数人での同時閲覧 | 高 |
| コミュニケーション | タスクへのコメント・メンション、Webhook連携 | 中 |
| 権限・組織管理 | メンバー招待・ロール設定、アクセス権限管理 | 高 |
| 可視化・レビュー | チーム進捗ダッシュボード、スプリント管理 | 低 |

---

## 3. P2P同期方式

### 3-1. 採用方式：iroh

| 方式 | シンプル | リアルタイム | オフライン | セキュリティ | 採用 |
|------|----------|--------------|------------|--------------|------|
| iroh | ◎ | ◎ | ○ | ◎ | ★ 採用 |
| mDNS単体 | ◎ | ◎ | ○ | △ | LAN限定のため却下 |
| libp2p | △ | ◎ | ◎ | ◎ | 実装コスト高 |
| 共有フォルダ | ◎ | △ | ◎ | △ | リアルタイム性なし |

> irohはRustクレートとして提供されており、TauriのRustバックエンドにそのまま組み込める。通信はQUIC+TLSでデフォルト暗号化済み。

### 3-2. irohのセキュリティ

| 項目 | 状況 | 備考 |
|------|------|------|
| 通信暗号化 | ◎ デフォルトで有効 | QUIC + TLS |
| 相手の認証 | ◎ 公開鍵ベース | EndpointID（Ed25519） |
| リレー経由の盗聴 | ◎ 復号不可 | iroh公開リレーサーバー |
| 入室制限 | △ アプリ側実装が必要 | 部屋コード＋ホスト承認 |
| 暗号アルゴリズム | ○ 現時点で十分 | Ed25519 / X25519 / P-256 |
| ポスト量子暗号 | △ 未対応 | 現時点では実用上問題なし |

### 3-3. NAT越えの接続経路

| 優先度 | 接続方式 | 条件 |
|--------|----------|------|
| 1 | 直接接続 | お互いが直接到達可能な場合 |
| 2 | ホールパンチング | NAT越えが必要な場合 |
| 3 | リレーサーバー経由 | 上記2つが失敗した場合（自前サーバー不要） |

## 4. チーム参加フロー

### 4-1. 部屋コード方式

irohのticket機能を活用し、部屋コードによる招待を実装する。

1. ホスト（A）がチームを作成 → TopicID（32バイト）を生成
2. 招待コードを発行（例：`KASTRIX-X7K2-9PQR`）
3. メンバーがコードを入力 → 参加申請が飛ぶ
4. ホストが承認 → メンバーがTopicIDをsubscribe
5. 全員のタスクデータが自動同期開始

> コードに有効期限（例：1時間）を設けることで、漏洩リスクを低減する。

### 4-2. 招待コードの有効期限

HOSTが発行時に有効期限を選択できる。デフォルトは1時間。

| 期限 | 用途 |
|------|------|
| 15分 | すぐ使ってもらう場合 |
| 1時間（デフォルト） | 通常の招待 |
| 24時間 | 後で使ってもらう場合 |
| 無期限 | 信頼できる相手（漏洩リスクあり） |

> 無期限コードを発行する際は「漏洩した場合は手動で無効化してください」という注意を表示する。

**発行済みコードの管理（Settings > チーム > 招待コード一覧）**

HOSTは発行済みコードを一覧で確認し、手動で無効化できる。漏洩時の緊急対応に使う。

```
KASTRIX-X7K2-9PQR  残り43分  [無効化]
KASTRIX-AB3C-7XYZ  残り2時間 [無効化]
```

### 4-4. 認証方式：コード＋ホスト承認の併用

| パターン | 概要 | 採用 |
|----------|------|------|
| コードのみ | コードを知っていれば入室可能 | ✗ |
| ホスト承認のみ | 申請→承認のフロー | ✗（ホストオフライン問題） |
| コード＋ホスト承認 | コード入力後にホストが承認 | ★ 採用 |

### 4-5. 共同ホスト（CO-HOST）

ホストがオフラインでも承認できるよう、承認権限を複数人に持たせる。

| ロール | 権限 |
|--------|------|
| HOST | 全権限、ロール変更、チーム解散、ブロック |
| CO-HOST | メンバー承認・キック、タスク管理 |
| MEMBER | タスクの閲覧・編集のみ |

> ホストが退出した場合は次のCO-HOSTに自動移譲する。権限変更もOperationとして全員に同期される。

### 4-6. キックとブロック

キックとブロックを分けて設計する。

| アクション | 権限 | ユースケース | 再参加 |
|-----------|------|-------------|--------|
| キック | CO-HOST以上 | 一時的な除外、誤追加 | HOSTが許可すれば可能 |
| ブロック | HOST限定 | 悪意あるメンバー | 新コードでも拒否 |

**キック・ブロックのOperation**

キック・ブロックは通常のOperationと同じ仕組みで配信する。ブロックOperationは`priority: high`フラグを立て、他のOperationより優先して適用する。

```json
{
  "type": "member_block",
  "priority": "high",
  "payload": { "targetId": "EndpointID_of_E" },
  "memberId": "yuki",
  "signature": "...",
  "timestamp": "2026-03-11T10:30:00Z"
}
```

**membersテーブルのstatus**

```
status: "active" | "pending" | "kicked" | "blocked"
```

**ブロック解除**

HOSTがブロック解除 → statusを`blocked` → `kicked`に変更 → 新しい招待コードを発行すれば再参加可能。既存の参加フローを流用する。

**注意事項（UI上に表示）**

> ⚠️ Kastrixを再インストールすることでブロックが回避される可能性があります。ホスト承認で対応してください。

### 4-7. 承認フロー

| 項目 | 決定内容 |
|------|---------|
| 通知先 | HOST + CO-HOST全員 |
| 承認期限 | なし（ずっとpending） |
| キャンセル | 承認前のみ申請者が取り消し可能 |
| 複数申請 | まとめてリスト表示 |

**UIイメージ（Inbox）**

```
参加申請 [2件]
  E（EndpointID: abc...）  申請日時: 2026-03-11 10:30  [承認] [拒否]
  F（EndpointID: def...）  申請日時: 2026-03-11 10:31  [承認] [拒否]
```

**キャンセルのOperation**

申請者自身が取り消す場合は`member_cancel`Operationを生成する。自分自身のOperationのため署名も自然に付けられる。

```json
{
  "type": "member_cancel",
  "payload": { "targetId": "EndpointID_of_E" },
  "memberId": "E自身のEndpointID",
  "signature": "..."
}
```

**同時承認の競合**

HOSTとCO-HOSTが同時に承認した場合、2つ目の承認Operationは「すでにactive」として無視される（冪等性）。

## 5. 更新処理・同期

### 5-1. Operationベースの差分同期

変更全体ではなく差分データ（Operation）のみをやり取りする。SQLiteにはOperationを全件積み上げる。

| フィールド | 型 | 説明 |
|------------|-----|------|
| id | TEXT | UUID |
| seq | INT | 連番（順序保証） |
| prev_id | TEXT | 1つ前のOperation ID（チェーン） |
| type | TEXT | task_update / member_join / permission_change など |
| payload | TEXT | 変更内容（JSON） |
| member_id | TEXT | 操作したメンバーのEndpointID |
| signature | TEXT | Ed25519署名 |
| timestamp | TEXT | NTP時刻 |
| ts_source | TEXT | "ntp" or "local" |
| synced | INT | 配信済みフラグ（0 or 1） |

### 5-2. タイムスタンプ管理

**取得タイミング：ボタンを押した瞬間（同期モードに関わらず統一）**

```
ボタンを押す
  ↓
NTPを叩いてタイムスタンプ取得（pool.ntp.org）
  ↓
Operationを生成（タイムスタンプ確定）
  ↓
ローカルSQLiteに保存
  ↓
  ┌─────────────────┐
自動モード          手動モード
即配信          Pushまで待機
```

| 状況 | ts_source | 衝突時の扱い |
|------|-----------|-------------|
| オンライン時 | ntp | 信頼度：高、衝突時に優先 |
| オフライン時 | local | 信頼度：低、ntpに負ける |
| local vs local | local | ユーザーに確認ダイアログ |

### 5-3. gossip伝播

iroh-gossipを使い、OperationをチームにPub/Sub方式で配信する。

- TopicID = チームID（招待コードから生成）
- Aが更新 → TopicIDにpublish → 同じTopicIDをsubscribeしているB,C,Dに自動配信
- 受け取ったノードが次のノードにリレーする（サーバー不要）

### 5-4. オフライン復帰

オフライン中のメンバーが復帰した際は、不在中のOperationを一括受信して順番に適用する（イベントソーシング）。

| 衝突パターン | 解決策 |
|-------------|--------|
| ntp vs ntp | タイムスタンプが新しい方を採用（Last Write Wins） |
| ntp vs local | ntpを優先 |
| local vs local | ユーザーに確認ダイアログを表示 |
| 衝突なし | そのまま順番に適用 |

## 6. 同期モード設定

ユーザーがSettings画面で同期タイミングを選択できる。この設定はローカルにのみ保存し、チームには同期しない。

| モード | 挙動 | 向いてるユーザー |
|--------|------|-----------------|
| 自動同期（デフォルト） | 操作したら即チームに配信 | Notionのような感覚で使いたい人 |
| 手動同期 | Pushボタンを押したときだけ配信 | 作業中の状態を共有したくない人 |

> 手動同期モード時はサイドバーに未配信Operation数をバッジ表示し、Pushボタンで一括送信する。

## 7. DBの設計

### 7-1. テーブル構成

| テーブル | 役割 |
|----------|------|
| operations | 全変更履歴。これが唯一の真実。 |
| tasks | operationsを積み上げた現在の状態（再構築可能） |
| members | チームメンバーと権限 |
| sync_state | 各メンバーとの同期状態管理 |

> operationsが日記、tasksが今日の状態。operationsさえ完全なら、tasksはいつでも再構築できる。

### 7-2. SQLite整合性対策

| 優先度 | 対策 | 効果 |
|--------|------|------|
| 必須 | トランザクション | 中途半端な書き込みを防ぐ |
| 必須 | WALモード（PRAGMA journal_mode=WAL） | 読み書き競合耐性を向上 |
| 重要 | Operation連番 | 順序の狂いを防ぐ |
| 推奨 | チェックサム突合（定期） | ズレを自動検知・修復 |

## 8. DB暗号化

### 8-1. 採用方式

| 層 | 対策 | ライブラリ |
|----|------|-----------|
| DBファイル | SQLCipherでAES-256暗号化 | sqlcipher |
| 暗号化キー | OSキーストアに保存 | keyring（仕様書記載済み） |
| 通信 | QUIC+TLSで暗号化済み | iroh（デフォルト） |
| 起動中 | OSのプロセス分離に委ねる | — |

### 8-2. 起動時のフロー

1. Kastrix起動 → OSキーストアにDBキーを問い合わせ
2. キーあり（2回目以降）→ キーを取得 → SQLCipherでDB復号 → 起動完了
3. キーなし（初回）→ DBキーを新規生成 → OSキーストアに保存 → 新規DBを暗号化して作成
4. 終了時 → メモリ上のキーを破棄（DBは暗号化されたままディスクに残る）

### 8-3. Linuxフォールバック

Secret Service APIが使えない最小構成環境では、起動時にパスワード入力を求め、PBKDF2でDBキーを導出する。

## 9. 改ざん耐性

### 9-1. Operationへの署名

各OperationにirohのEndpointID（Ed25519）で署名を付与する。SQLiteを直接書き換えた場合、他メンバーのSQLiteと不一致になるため検知できる。

### 9-2. 多数決チェック

複数メンバーのSQLiteを突合し、不一致があれば警告を出す。1台が改ざんされても他メンバーの記録で弾ける。

### 9-3. 権限変更への攻撃対策

| 攻撃シナリオ | 対策 |
|-------------|------|
| 招待コードの漏洩 | ワンタイム＋有効期限＋ホスト承認の併用 |
| ホストPCへの侵入・権限変更 | 権限変更に複数CO-HOSTの承認を要求（多数決） |
| SQLiteの直接改ざん | 他メンバーとのチェックサム突合で検知 |
| なりすまし | irohの公開鍵暗号で不可能 |

> PC自体が乗っ取られるレベルの攻撃はOSのセキュリティ領域。Kastrixは多数決承認と操作ログの透明性で対応する。

## 10. バージョン管理

### 10-1. 基本ルール

以下の4つのルールを守ることで、ほとんどのケースを自然に解決できる。

| ルール | 内容 |
|--------|------|
| 1 | 新しいフィールドは必ずオプショナルにする |
| 2 | Operationに`version`フィールドを最初から入れる |
| 3 | 知らない`type`のOperationは無視してスキップ |
| 4 | 破壊的変更のときだけ強制アップデートを促す |

### 10-2. Operationのversionフィールド

```json
{
  "version": "1.0",
  "type": "task_update",
  "payload": { ... }
}
```

### 10-3. 受け取り側の処理

```
知っているversionなら処理
知らないversionなら「アップデートが必要です」と表示してスキップ
知らないtypeならスキップ
```

### 10-4. チームでバージョンが混在する場合

```
A: v1.1（最新）
B: v1.0（古い）
  ↓
AがB に「アップデートしてください」と通知
  ↓
Bがアップデートするまで破壊的変更のOperationはスキップ
```

開発チーム向けツールのため、Slackなどで「アップデートしてください」と一言伝えれば解決するケースがほとんど。

### 10-5. 変更の種類別の対応方針

| 変更の種類 | 対応 |
|-----------|------|
| フィールド追加 | オプショナルにして後方互換を維持 |
| フィールド削除・変更 | versionを上げて古いクライアントに警告 |
| Operation typeの追加 | 知らないtypeはスキップ |
| 破壊的変更 | 強制アップデートを促す |

> 複雑なマイグレーション機構は後から追加できる。最初はシンプルに、問題が出たら足す方針で進める。

## 11. 将来の拡張案

現時点では必須ではないが、長期運用で必要になった場合に追加する機能。

### 11-1. スナップショット（データ肥大化対策）

Operationを積み続けると新規参加者への初回同期が遅くなる可能性がある。その時点の状態をまるごと保存するスナップショットで対応する。

小規模チームや運用期間が短い場合は全Operation受け取りでも問題ない。「初回同期が遅い」と感じたタイミングで追加すれば十分。

```json
{
  "snapshotId": "snap-2026-01-01",
  "version": "1.0",
  "timestamp": "2026-01-01T00:00:00Z",
  "tasks": [ ... ],
  "members": [ ... ],
  "lastOperationSeq": 5420
}
```

実装する場合は参加時に「スナップショットから始める / 全履歴から始める」を選択できるようにする。

## 12. 実装フェーズ

| Phase | 内容 | 目標 |
|-------|------|------|
| Phase 1 | iroh接続 + 部屋コード参加 + Operation基本同期 | 最小限の同期が動く状態 |
| Phase 2 | NTPタイムスタンプ + WAL + 連番 + ホスト承認 | 整合性と認証の基盤 |
| Phase 3 | SQLCipher暗号化 + keyring連携 + チェックサム突合 | セキュリティ強化 |
| Phase 4 | 同期モード設定 + CO-HOST + 衝突ダイアログ | UXと権限管理の完成 |
| Phase 5 | キック・ブロック + バージョン管理 + 強制アップデート通知 | 運用品質の向上 |
